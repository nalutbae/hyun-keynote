기억의 본질

AI 에이전트를 오래 쓸수록 기억이 쌓입니다. 이때 모든 것을 하나의 memory에 넣으면 장기 기억은 금방 무거워지고, 정작 매번 필요한 기준은 묻힙니다. 반대로 아무 곳에도 정리하지 않으면 에이전트는 같은 설명을 계속 다시 요구합니다.

핵심은 "나중에 다시 쓸 때 어떤 형태로 꺼내야 하는가"입니다.

기억의 다섯 가지 종류

AI 개인비서가 기억해야 하는 것은 하나가 아닙니다. 반복 선호, 역할 경계, 팀 규칙, 누적 지식, 대화 기록 — 이 다섯 가지는 각각 다른 곳에 두어야 합니다.

모두를 하나의 memory에 넣으면 금방 무거워지고, 아무 곳에도 두지 않으면 매번 같은 질문을 반복하게 됩니다.

에르메스 에이전트 기초 가이드 · KEYNOTE 1-4

세 가지 기억이 다루는 것

session

지금 이 대화의 맥락

  • 현재 대화에서만 유지
  • 세션이 끝나면 사라짐
  • 긴 대화에서는 compaction 대상
  • 작업 중인 파일, 방금 한 결정
memory

앞으로도 계속 쓸 기준

  • 세션을 넘어 계속 유지
  • 사용자 선호, 반복 규칙
  • 적고 강해야 한다
  • 너무 길어지면 판단이 흐려짐

session과 memory의 차이

session은 지금 이 대화에서만 유지되는 맥락입니다. 대화가 끝나면 사라집니다. 반면 memory는 세션을 넘어 계속 유지되는 장기 기준입니다.

핵심은 "이 정보가 다음 대화에서도 필요한가?"입니다. 그렇다면 memory에 두고, 아니라면 session에만 둡니다. profile은 에이전트 자체의 역할과 도구 설정입니다.

네 가지 기억 위치

SOUL.md, AGENTS.md, USER.md, skill은 각각 다른 질문에 답합니다. "누구인가", "어떻게 일할 것인가", "사용자가 무엇을 원하는가", "반복되는 절차는 무엇인가" — 이 네 질문이 섞이면 유지보수가 어려워집니다.

예를 들어 "뽀동이는 콘텐츠 퀄리티를 책임지는 햄스터다"는 SOUL.md에, "WikiDocs 작업 전에는 H1, 이미지 경로, 본문 링크를 검증한다"는 AGENTS.md나 skill에 들어가야 합니다.

기억층별 분리

각 기억층은 역할이 다릅니다. memory는 항상 필요한 짧은 기준, shared-memory는 팀 공통 작업 원본, skills는 반복 절차, Obsidian은 누적 지식 본체, session_search는 과거 회상, OpenViking/RAG은 검색 인덱스입니다.

넣으면 위험한 것을 보면 패턴이 보입니다. 긴 로그, 개인 메모, 미검증 원문, 정제되지 않은 더미 — 이런 것들은 다른 곳에 두어야 합니다.

기억 이주

다른 시스템에서 Hermes로 넘어올 때 기억을 그대로 복사하면 안 됩니다. 기억 이주는 기준 재구성입니다. 과거 설정을 현재 기억 구조에 맞게 나누어 재배치해야 합니다.

하비는 기억 오케스트레이터다

1-4-2. 기억 오케스트레이션과 강화10 / 18

기억 오케스트레이터

하비는 요청을 받아 다른 에이전트에게 넘기는 라우터에 머물지 않습니다. 모든 기억층을 보고 어디에 무엇을 둘지 판단하는 기억 오케스트레이터입니다.

이 역할이 없으면 에이전트 팀은 정보를 많이 갖고 있어도 매번 같은 질문을 반복하게 됩니다. 누가 무엇을 기억해야 하는지, 어떤 원본을 먼저 봐야 하는지 판단되지 않기 때문입니다.

판단 질문

하비가 기억 오케스트레이터로 판단할 때 핵심 질문은 "이걸 기억할까?"가 아니라 "어떤 층에 남길까?"입니다. 이 표가 그 판단 기준입니다.

실제 운영에서는 이 분류가 자동으로 되지 않습니다. 하비가 잘못 판단하면 작업 로그가 memory에 들어가 장기 기억이 지저분해지거나, 공통 규칙이 개인 profile에만 있어 에이전트마다 기준이 달라집니다.

에르메스 에이전트 기초 가이드 · KEYNOTE 1-4-2

memory와 RAG는 목적이 다르다

memory

항상 필요한 장기 기준 주입

  • 크기를 작게 유지한다
  • 선호, 규칙, 역할 기준
  • 너무 길어지면 판단이 흐려진다
  • "중요하니까 memory에 넣자"는 잘못된 판단
OpenViking / RAG

필요한 지식 검색과 회수

  • 크기를 크게 확장 가능
  • 문서, 노트, 결정 기록, 지식 조각
  • 잘못 회수하면 근거가 왜곡된다
  • "나중에 찾아야 하니까 RAG 대상으로 만들자"

memory와 RAG의 차이

memory와 RAG를 같은 것으로 보면 설계가 흔들립니다. memory는 적고 강해야 하고, RAG는 넓고 회수 가능해야 합니다.

"중요하니까 memory에 넣자"와 "나중에 찾아야 하니까 RAG 대상으로 만들자"는 다른 판단입니다. 둘은 대체 관계가 아니라 보완 관계입니다.

에르메스 에이전트 기초 가이드 · KEYNOTE 1-4-2

RAG 회수층을 붙일 때의 기준

항상 필요한 기준

memory에 둔다

항상 주입해야 할 짧은 기준은 RAG가 아닌 memory가 맡는다

넓게 검색할 지식

Obsidian/RAG 대상으로 본다

크고 변하는 지식은 검색/회수 대상으로 둔다

회수 품질

질문 세트로 검증한다

느낌이 아니라 검색 적중률과 근거 품질로 평가한다

보호 정보

RAG 대상에서 제외하거나 보호한다

내부 경로, 토큰, 계정 정보는 검색 인덱스에 넣지 않는다

RAG 판단 질문

  • 지식이 너무 커서 memory에 넣기 어려운가? → RAG 필요
  • 자주 바뀌거나 계속 쌓이는가? → RAG 필요
  • 검색해서 일부만 가져와도 충분한가? → RAG 필요
  • 항상 모든 대화에 주입해야 하는가? → memory가 맞다

RAG 회수층 기준

OpenViking/RAG 같은 기억 강화층은 memory를 대체하지 않습니다. 항상 주입해야 할 짧은 기준은 memory에 두고, 크고 변하는 지식은 검색/회수 대상으로 둡니다.

핵심은 "프롬프트를 길게 만드는 것"이 아니라 "필요한 순간에 맞는 지식을 꺼내는 것"입니다.

컨텍스트 흐려짐과 handoff

긴 대화는 편하지만 위험합니다. 앞에서 합의한 기준, 완료한 작업, 남은 범위가 계속 쌓이면 어느 순간 흐려집니다.

핵심 메시지

AI 에이전트의 장기 기억은 하나의 저장소로 해결되지 않습니다. 짧은 기준, 팀 공통 규칙, 반복 절차, 프로젝트 지식, 과거 대화, 검색 인덱스가 각자 다른 역할을 가져야 합니다.

좋은 구조는 에이전트가 모든 것을 혼자 기억하게 만드는 것이 아니라, 사람과 에이전트가 함께 관리한 지식을 필요한 순간에 꺼낼 수 있게 만드는 것입니다.
