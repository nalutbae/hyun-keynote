핵심 정의

Hermes Agent를 다른 도구와 비교하기 전에 가장 중요한 질문은 "무엇이 더 좋나"가 아니라 "어떤 일을 맡길 것인가"입니다.

코드 작업이 중심이면 Claude Code/Codex, 업무 흐름이 중심이면 Hermes Agent가 먼저입니다.

Hermes Agent의 네 가지 실행 방식

01

CLI Chat

터미널에서 직접 대화. 설치 확인, 기본 검증, 간단한 질문에 적합

02

Gateway

Slack/Telegram/Discord에서 always-on으로 요청을 받아 처리

03

Cron

정해진 시간에 자동 실행. 조사, 보고, 점검을 반복 작업으로 운영

04

Delegation

역할형 에이전트에 작업을 위임. 조사/정리/실행/이미지 등 분업

네 가지 실행 방식

Hermes Agent는 한 가지 방식으로만 동작하지 않습니다. CLI로 시작해서 Gateway로 always-on 운영, Cron으로 자동화, Delegation으로 역할 분업까지 확장됩니다.

중요한 것은 이 네 가지가 개별 기능이 아니라 하나의 운영 흐름으로 이어진다는 점입니다.

기억과 자기 개선

Hermes Agent와 다른 에이전트를 비교할 때 가장 크게 느껴지는 차이는 기억 구조와 자기 개선 루프입니다.

WikiDocs 발행에서 이미지 경로 검증을 놓쳤다면 그건 단순한 실수가 아니라 Skill 보강 후보가 됩니다. 매번 새로 똑똑해지는 것이 아니라 사용자의 업무 방식과 실패 패턴을 기준/절차/검증 루프로 누적합니다.

비교 표

각 도구의 주 용도와 강점이 다릅니다. Claude Code는 코딩, Codex는 코드 수정, OpenClaw는 로컬 체감형 자동화, Hermes Agent는 업무 흐름 조율에 강합니다.

Hermes Agent는 경쟁자라기보다 어떤 일을 어떤 실행 방식에 맡길지 결정하는 상위 운영 흐름에 가깝습니다.

Hermes Agent가 적합한 경우

채널 섹션 전환

Hermes Agent는 하나의 앱 화면에서만 쓰는 도구가 아닙니다. 여러 채널에서 호출되는 AI 개인비서 운영 환경입니다.

처음에는 모델이나 프롬프트만 보게 되지만, 실제 업무 자동화에서는 "어디서 부를 것인가"가 중요해집니다.

Gateway와 채널

Gateway는 채널이 아니라 운영 축입니다. process가 살아 있어도 메시지가 안 오거나, delivery가 안 되거나, 권한 문제가 있을 수 있습니다.

gateway status OK ≠ 운영 완료. process, 인증, delivery target, 로그를 함께 봐야 합니다.

채널을 열기 전에 트리거 규칙을 정한다

01

멘션 또는 명시 호출

사람 대화와 봇 대화를 분리한다. 모든 메시지에 반응하는 봇은 가장 빨리 피로감을 만든다

02

봇별 역할 이름

조사형/정리형/실행형/이미지형 응답이 섞이지 않도록 역할을 명확히 한다

03

사람 대화 중 자동 멈춤

멀티봇 소음을 줄이기 위해 사람이 대화 중이면 자동 응답을 멈춘다

04

허용 사용자/채널

원하지 않는 실행 요청을 막기 위해 채널별 허용 범위를 둔다

05

전달 채널 분리

결과 전달 채널과 작업 채널을 구분하면 cron/delivery 실패를 찾기 쉽다

06

처음에는 하나만

주 채널 하나를 안정화한 뒤 확장하는 편이 좋다
