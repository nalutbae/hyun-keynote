왜 챗봇 하나로는 부족한가

챗봇은 질문에 답하는 데 강하다하지만 실제 업무는 답변 뒤에 조사, 정리, 실행, 검증, 기록이 이어진다이 단계가 붙는 순간 AI는 "좋은 답변을 주는 도구"에서 "반복 업무를 함께 굴리는 운영 구조"로 바뀐다

왜 챗봇만으로는 부족한가

처음에는 AI가 답변만 잘하면 충분해 보입니다. 하지만 업무가 반복되면 금방 문제가 생깁니다. 지난번에 어디까지 확인했는지, 조사 결과와 해석이 섞이고, 실행이 필요한데 설명만 길어지는 현상을 겪게 됩니다.

이때 필요한 것은 더 긴 프롬프트가 아니라 역할과 기록의 구조입니다.

에르메스 에이전트 기초 가이드 · KEYNOTE 1-3-1

같은 요청, 다른 처리

챗봇 관점

한 번의 대화 안에서 끝낸다

  • 검색 결과를 요약한다
  • 문장을 자연스럽게 고친다
  • 가능한 방법을 설명한다
  • 현재 대화에서만 찾는다
  • 내용을 보기 좋게 요약한다
AI 팀 관점

요청을 운영 단위로 바꾼다

  • 조사 범위, 근거, 반례, 후속 질문을 나눈다
  • 독자, 목적, 구조, CTA, 발행 채널을 맞춘다
  • 도구, 권한, 스케줄, 실패 시 확인 순서를 정한다
  • 세션 기록, 문서, 원본 기준을 확인한다
  • 읽는 사람에게 필요한 것과 내부 정보를 구분한다

비교

같은 요청도 챗봇 관점과 AI 팀 관점에서는 완전히 다르게 처리됩니다. 챗봇은 질문에 답하는 데 강하고, AI 팀은 요청을 해석하고 역할을 나누고 결과를 묶는 데 강합니다.

에르메스 에이전트 기초 가이드 · KEYNOTE 1-3-1

실제 업무에서 보이는 다섯 가지 차이

근거

조사 결과가 충분하지 않은데 글이 먼저 매끄러워진다

조사형 에이전트가 근거와 반례를 확보해야 정리형이 제대로 글을 쓸 수 있다

실행

실행이 필요한 일인데 설명만 길어진다

파일 수정, 검증, 배포 같은 상태 변경은 실행형 에이전트가 맡아야 한다

기록

기준이 대화 속에 묻히고 다시 찾지 못한다

memory와 Skill로 반복 기준을 남기면 매번 처음부터 설명하지 않아도 된다

구조

이미지와 표가 필요한데 본문만 늘어난다

디자인 에이전트가 시각 메시지를 잡아야 내용이 제대로 전달된다

다섯 번째: 모든 역할이 따로 논다

  • 문제 조사, 정리, 실행, 기록이 한 대화 안에서 섞인다
  • 해결 PM 에이전트가 전체 흐름을 통합하고 최종 상태를 보고한다
  • 기준 위임 기준, 최종 검수, 보고 형식이 명확해야 한다

실무에서의 차이

하나의 에이전트에게 모든 일을 맡기면 다섯 가지 문제가 생깁니다. 근거가 약한데 글이 먼저 매끄러워지고, 실행이 필요한데 설명만 길어지고, 기준이 대화 속에 묻힙니다. 역할 분리는 장식이 아니라 실패 지점을 보이게 만드는 안전장치입니다.

역할 분리의 이유

AI도 하나의 에이전트에게 모든 일을 맡기면 조사 결과가 충분하지 않은데 글이 먼저 매끄러워지고, 실행이 필요한데 설명만 길어지며, 기록이 대화 속에 묻힙니다. 역할 분리는 선택이 아니라 운영의 기본입니다.

여섯 가지 실무 역할

PM / 메인 창구

요청 해석, 우선순위, 위임, 최종 통합

사용자 앞에서 요청을 받고, 역할을 나누고, 결과를 묶어 보고하는 입구

조사 (Research)

근거 수집, 사실 확인, 반례 탐색

공식 문서와 세션 기록을 대조하고 근거와 반례를 모으는 역할

정리 (Structure)

문서화, 구조화, 보고서 정리

독자 흐름에 맞게 재구성하고 목차와 CTA까지 잡는 역할

기획 (Planning)

요구사항, 기능 흐름, 제품 문서화

사용자 문제와 기능 흐름을 정의하고 판단 기준을 세우는 역할

실행 / 개발 (Dev)

파일 수정, 명령 실행, 검증, 배포

상태 변경을 안전하게 처리하고 검증 결과와 rollback 기준을 남긴다

디자인 (Design)

이미지 기획, 시각 콘셉트, 썸네일

핵심 문장을 시각 메시지로 바꾸고 가독성과 채널을 고려한다

역할 이름이 라우팅 규칙이 되는 방식

요청 예시

실무 언어로 요청하면 역할이 분리된다

  • PM 에이전트에게 요청을 전달한다
  • 조사 에이전트가 근거와 반례를 수집한다
  • 정리 에이전트가 독자 흐름을 잡는다
  • 디자인 에이전트가 이미지 방향을 설정한다
  • 개발 에이전트가 파일을 수정하고 검증한다

라우팅 규칙

역할 이름은 귀엽게 부르기 위한 장식이 아니라 라우팅 규칙입니다. "조사 에이전트에게 확장조사를 맡긴다"고 말하면 그 안에 역할과 기대 산출물이 이미 들어 있습니다.

실제 운영에서만 의미 있는 세부값이나 사적인 맥락은 빼고, 어떤 역할이 왜 필요했고 어떤 실패를 줄였는지만 남깁니다.

운영 루프

Hermes가 운영 시스템인 이유는 대화에서 끝나지 않고 실행, 기록, 재사용까지 이어지기 때문입니다. 좋은 답변을 말하는 것과 실제로 확인하고 고치고 남기는 것은 다릅니다.

아래 운영 층별 확인 질문은 각 층에서 무엇을 점검해야 하는지 보여줍니다. 창구, 역할, 상태, 도구, 기록이 모두 연결되어야 운영이 됩니다.

한 번의 요청이 자산이 되려면

좋은 답을 말하는 것과 실제로 확인하고 고치고 남기는 것은 다르다Hermes는 대화를 잘하는 AI에서 끝나는 것이 아니라 실행 → 검증 → 기록 → 재사용으로 이어져야 운영 시스템이 된다이게 빠지면 대화는 휘발되고, 시스템은 반복 학습 대신 반복 삽질을 하게 된다

기록과 재사용

Hermes를 운영 시스템으로 봐야 하는 이유는 바로 여기에 있습니다. 대화만 잘하는 시스템은 한 번의 요청이 휘발됩니다. 하지만 기록과 재사용이 연결되면 매번 같은 실수를 반복하지 않아도 됩니다.

이것이 챗봇과 운영 시스템의 가장 큰 차이입니다.

핵심은 "여러 AI를 만들었다"가 아니라, 하나의 반복 업무가 안정적으로 굴러가고, 피드백이 기억과 Skill로 남고, 다음 역할을 붙였을 때 품질이 더 좋아지는지 확인하는 것입니다.
